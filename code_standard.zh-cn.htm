==CMP框架衍生代码标准 （20150706）

 

=文档目标

让MG框架的衍生代码规范化

 

=框架的中心设计思路 （ http://cmpTech.info）

HTTP入口 index.php?class=$className&method=$methodName&param=XXXXX

其中：

1、参数名 class/method/param可以简化为 _c/_m/_p （即CMP的来源，我们把接口归一化）

2、_p默认为 json化（且一般用POST方法传入），框架兼容但不欢迎用GET显式传入

3、参见下面【URL重写】该段描述。

 

Class $className

{

public function $methodName($param){

//默认param即为_p，但如果 _p 为空，框架要把 $_REQUEST 作为param传入此处。** 约定

}

}

=关于URL重定向

为了使得访问借口变得优变一点，通过修改 yaml、htaccess、apache配置等把

index.php?class=$className&method=$methodName

重定向

,$className,$methodName.web （通常为 WebXXX）

,$className,$methodName.json （通常为 WebXXX、ApiXXX类、JsonXXX类）

$className,$methodName.api （更通用模式）

 

特别地，伪静态 XXX.shtml有特别处理，将跳过php逻辑处理，直接进入模板 shtml.XXX.html

 

=其它类命名

LgcYYY  Lgc for Logic，即业务逻辑层

AppXXX   应用层级的一些对象类

OrmZZZ   Orm for Object related model，主要是比较纯粹的数据库封装，尽量不做过多的逻辑。而且 目前主要采用 RedBeanPHP 的封装方法。

 

* 以上直观理解为 Api层=>（参数简单过滤、权限简单校验）=> Lgc逻辑层 （含深度校验）=>计算、操作 业务对象（AppXXX）及 数据对象（OrmZZZ）。 但由于历史原因，部分 App/Orm对象可能不是分得那么清晰。

 

=命名规范

注意除历史错误俗称的原因以外，类名尽量用大驼峰法，比如

WebBusiness

ApiBusinessReport

ApiBusinessTools

等

 

函数，如果是接口类的public函数，一般也用大驼峰，例：

public function WebTest->TestAbc()

 

如果是非Api接口类的public函数 或protected可继承的，一般用小驼峰，例：

Class LgcTest 的public function abcDef();

 

如果是私有函数，首字母前加下划线，并使用小驼峰

Private function _cdef();

 

l       由于历史原因，少量已经用得比较多错误明名函数有可能一时未能全部改回来

l       由于部分旧代码可以接受非驼峰命名（如下划线法），部分涉及全大定的专用名词缩写在很难阅读的情况下也会使用下划线法 如 ACE_ACCT（当然用 AceAcct也是可以）

 

 

=参考资料：《命名法》

1. 匈牙利法：

开头字母用变量类型的缩写，其余部分用大驼峰。

比如 iXXX, iptYYY, strZZZ

 

2. 小驼峰式：(little-camel-case)

第一个单词首字母小写，后面其他单词首字母大写。

For example: string firstName = string.Empty;

 

3. 大驼峰式：(big-camel-case) * 有时又叫Pascal法，其实两者基本差不多

每个单词的第一个字母都大写;

For example：string FirstName = string.Empty;

 

4. C语言法

全小写混合，下划线随意

 

5. 下划线法（php与perl函数常用）

单词之间用下划线间隔，单词本身大小写看情况，但通常是全小写
